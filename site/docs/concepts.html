Concepts

The mental model behind Artifact — the trust bubble, sites, sessions, and why all primitives live on the server.

The trust bubble

Every company already has a trust bubble: the set of infrastructure reachable only by verified employees. Workday, Confluence, GitHub Enterprise, and your internal dashboards all live there. Getting past the perimeter means passing your SSO.

Artifact lives entirely inside this bubble. Because every HTTP request is already an authenticated employee, Artifact can expose powerful primitives without any API keys, CORS configuration, or per-user auth code. The trust that your company already established flows directly into every app.

🚨

Artifact is not designed for public internet exposure. Never put an Artifact server on a public ingress. It is designed exclusively for internal deployment behind corporate SSO or a zero-trust proxy.

Sites

A site is the unit of deployment. Each site:

  • Has a name (e.g. lunch-poll) that maps to a subdomain (lunch-poll.artifact.corp.com)
  • Contains a set of static files (HTML, CSS, JS, images)
  • Has its own isolated document collections and KV store
  • Has a recorded owner (the employee who first deployed it)

You can have thousands of sites on a single Artifact server. They share the database and storage infrastructure but are logically isolated.

Deploys

Each artifact deploy creates a new deploy object. Artifact:

  1. Uploads all files to object storage under a deploy ID prefix
  2. Writes a manifest JSON listing all files and their content hashes
  3. Atomically updates a manifest pointer in the database

The atomic pointer flip means a site is never served in a half-deployed state. Until the pointer flips, the old version continues serving. There is no downtime during a deploy.

Artifact keeps an LRU cache of recently-accessed deploy IDs in memory so manifest lookups are O(1) for hot sites.

Sessions

When a user authenticates via OIDC, Artifact:

  1. Exchanges the auth code for tokens with the IdP
  2. Creates a session record in the database (table: sessions)
  3. Sets a signed, opaque artifact_session cookie

The session stores the user's email, name, and group memberships. Sessions persist across server restarts (they're in the DB). Session TTL defaults to 12 hours and is refreshed on activity.

In dev mode, sessions are replaced by a fake dev@localhost identity injected on every request.

In header-trust mode, identity comes from headers set by an upstream IAP (e.g. X-Forwarded-Email). No database sessions are used.

Anatomy of a request

Every request to Artifact flows through:

  1. Auth middleware — validates session cookie / header / OIDC state; rejects unauthenticated requests with a redirect to the login page
  2. Subdomain router — extracts the site name from the Host header (e.g. lunch-poll.artifact.corp.com → site lunch-poll); requests to the apex domain show the site directory
  3. Route handler:
    • GET /_/* → serve static file from object storage (with ETag / conditional GET)
    • POST /api/v1/db/* → database CRUD
    • GET /ws → upgrade to WebSocket
    • POST /api/v1/ai/* → proxy to upstream LLM
    • etc.
  4. Audit logging — every API call is recorded in the audit log with user, site, action, and timestamp

Why no custom backends?

You might wonder: why can't I run server-side code per site? The answer is about operational simplicity and security. Custom backends mean:

  • Per-site servers that need on-call ownership
  • Security reviews for every throwaway tool
  • Dependency upgrades across hundreds of repos
  • Unknown network egress from arbitrary code

Artifact's bet is that a fixed set of primitives covers 95% of internal apps when you're inside the trust bubble. The things that typically require a backend — auth, database writes, AI calls, file storage — are provided with zero configuration. When you hit a wall, the answer is almost always "combine two SDK primitives."

For the remaining 5%, Artifact sites can call other internal services (your existing APIs) from the browser, since they're in the same trust bubble.

Storage backends

Artifact supports three object storage backends, selectable in artifact.yaml:

BackendConfig keyBest for
Local diskstorage.type: localDev / single VM
S3 / MinIOstorage.type: s3AWS, self-hosted MinIO
GCSstorage.type: gcsGoogle Cloud

Database backends

BackendConfig keyBest for
SQLitedb.type: sqliteDev, single-server prod
PostgreSQLdb.type: postgresProduction, multi-replica

Multi-replica scaling

Artifact scales horizontally with:

  • External Postgres as the shared database
  • S3 / GCS as the shared object store
  • NATS for WebSocket pub/sub across replicas

Without NATS, WebSocket events only reach clients connected to the same replica. With NATS, any replica that receives a DB write broadcasts it to all connected clients cluster-wide.
