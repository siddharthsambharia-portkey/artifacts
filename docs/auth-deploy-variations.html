Artifact · agent deploy auth · no-token design

Deploying a site from an agent — with no Artifact token

The goal: an employee says “deploy this” and a site goes live at name.artifacts.company.com. Constraint: Artifact issues no token of its own. The agent must prove identity using a credential that already exists.

✗ no Artifact token (no PAT, no minted token) ✓ skills are the only default format — CLI optional — MCP not required
The no-token rule changes the winner. With header-trust there’s already a shared credential the agent can carry — the proxy secret — so a plain curl from a skill works with no token, no login, no CLI, no MCP. With native OIDC there is no pre-existing shared credential and tokens are banned, so a headless agent has nothing clean to present — it’s pushed to fragile cookie reuse or the browser drag-and-drop. Header-trust is the natural fit for your setup.

Variation 1 Okta — Header-Trust fits the constraints

oauth2-proxy already does Okta SSO for browsers and Artifact trusts it via a shared secret (X-Artifact-Proxy-Auth, already enforced in header_trust.go). That secret is the credential the agent reuses — no token needed.

  1. One-time setup operator, once
    The proxy secret is placed where agents can read it (e.g. an env var on the dev box / CI). No per-user login, no Okta admin.
  2. Agent runs the deploy skill agent
    The skill is just a curl recipe. No CLI install, no MCP server.
  3. Request goes around the proxy, straight to Artifact agent
    Carries X-Artifact-Proxy-Auth (proves “inside the trust boundary”) + X-Email (self-asserted identity).
  4. Artifact accepts and serves the site
    In trust mode all employees are equal, so self-asserted identity is enough. → https://my-site.artifacts.company.com
Every deploy: just “deploy this” → URL. Zero clicks, zero tokens. The skill carries everything.

Variation 2 Okta — Native OIDC awkward without a token

Artifact is the Okta app and owns login. There is no proxy in front, so there is no pre-existing shared secret to lean on. Native OIDC’s natural credential is a token — which this design forbids.

  1. Agent runs the deploy skill agent
    It has no token and is not allowed to mint one.
  2. Only token-free option: reuse a browser session cookie fragile
    User signs in via Okta in a browser; the agent must scrape and replay the session cookie.
  3. Cookie expires → re-login fragile
    Short-lived sessions break the “zero-friction” promise; extracting cookies is brittle and agent-hostile.
  4. Fallback: human drag-and-drop in the web UI
    Reliable, but it isn’t the agent doing it — which defeats the point.
Net: with no token allowed and no proxy secret to borrow, there’s no clean headless path. Native OIDC wants a token; remove it and you’re left with cookies or manual upload.
the entire integration — what the skill teaches the agent
# Header-trust, no token. The skill bundles this curl. No CLI, no MCP.
curl -X POST https://artifacts.company.com/api/v1/deploy \
  -H "X-Artifact-Proxy-Auth: $ARTIFACT_PROXY_SECRET" \
  -H "X-Email: you@company.com" \
  -F "site=my-site" \
  -F "files=@index.html"

# → { "site": "my-site", "url": "https://my-site.artifacts.company.com", ... }

Need a CLI? No

The CLI is just sugar over the curl above. A skill that runs curl deploys a site with zero extra binaries to install or version.

Need MCP? No

MCP is one more transport wrapping the same HTTP call. If a skill can run curl, an MCP server for deploy is redundant.

Skills only? Yes

Skills are the single default format. artifact init drops a skills folder; the agent reads it and just works — the most Quick-like path.

The one honest tradeoff in the no-token design

Without a token, the shared proxy secret proves “this came from inside the company,” not “this is a specific person.” The email is self-asserted. That has a clear cost — and a clean upgrade path.

Fine for v1 (trust mode)
governance.mode: trust

All employees are equal — no ownership, no per-site permissions. Self-asserted identity is acceptable; the proxy already blocks outsiders. Stands up in an afternoon, skills-only.

What it does NOT give you (yet)
provable per-user identity

Anyone holding the secret can claim any email, so the audit log isn’t trustworthy and it won’t pass a security review. Provable identity is exactly what native OIDC / governed mode buys later — the Stage-2 upgrade.

one-time / browser agent step (silent) fragile / not recommended
